## 01.01 — Encoding and Character Sets

Every time a computer reads or writes text, it must convert bytes to characters and vice versa. The mapping between bytes and characters is called an **encoding**.

| Encoding | Byte size | Supports CJK? | Common use                |
|----------|-----------|---------------|---------------------------|
| ASCII    | 1 byte    | No            | Legacy, English-only      |
| UTF-8    | 1–4 bytes | Yes           | Web, Linux, Python 3+     |
| GBK      | 1–2 bytes | Yes           | Legacy Windows (China)    |
| UTF-16   | 2–4 bytes | Yes           | Windows internals, Java   |

> See also: [[01-编码原则#01.02 — UTF-8 Best Practices]]

### Common pitfall

On Windows with a Chinese locale, the terminal code page is often GBK (CP936). If you pipe CJK text through a tool expecting UTF-8, you get garbled output or a `UnicodeDecodeError`.

**Solution:** Explicitly set `PYTHONIOENCODING=utf-8` or use `chcp 65001` before running Python in cmd.

## 01.02 — UTF-8 Best Practices

1. **Always declare encoding** at the top of Python files: `# -*- coding: utf-8 -*-`.
2. **Use `encoding='utf-8'`** in every `open()` call.
3. **Avoid CJK in tool arguments** — pass them via config or environment variables instead.
4. **Test on the target OS** — what works on macOS (UTF-8 by default) may fail on Windows (GBK).
