# Rules — Programming Domain

Coding conventions and project-specific guidelines for software development tasks.

## Rule 1 — Build and run

1. **Always build from the correct working directory**: For Go projects with a `cmd` subdirectory, change into `cmd/<project>` before running `go build`.
2. **Check syntax before running**: Lint or compile first; do not execute code that has not been validated.
3. **Use version-controlled dependencies**: Prefer `go mod tidy` / `pip install -r requirements.txt` over ad-hoc `go get` / `pip install`.

## Rule 2 — Code modification

1. **Read the full file before editing**: Use `read_file` (or equivalent) to inspect the current content before making changes.
2. **Edit, don't rewrite**: Use targeted `edit_file` to change specific lines rather than writing the entire file — this preserves context and reduces merge conflicts.
3. **Run tests after every change**: Execute the project's test suite (or at minimum compile/validate) after any non-trivial modification.

## Rule 3 — Error handling

1. **Never swallow exceptions silently**: All error handlers should at minimum log the exception message and stack trace.
2. **Return actionable errors**: Error messages should state what went wrong and what the user/AI can do to fix it.
3. **Graceful degradation**: If a non-critical component fails, the system should continue functioning for the remaining features.
