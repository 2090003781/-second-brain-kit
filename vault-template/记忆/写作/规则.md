# Rules — Writing Domain

Guidelines for producing clear, maintainable, and reader-friendly content.

## Rule 1 — Structure

1. **Start with an outline**: Before writing a long document, list the main sections in a comment or scratchpad.
2. **Use semantic headings**: Heading levels should reflect the document hierarchy (H1 = title, H2 = sections, H3 = subsections). Do not skip levels.
3. **Keep paragraphs short**: Aim for 3–5 sentences per paragraph. One idea per paragraph.

## Rule 2 — Style

1. **Prefer active voice**: "The agent writes the file" over "The file is written by the agent".
2. **Use plain English**: Avoid jargon unless it is defined in context. If a technical term is necessary, link to its definition.
3. **Be concise**: Remove filler words ("basically", "actually", "in order to" → "to").

## Rule 3 — Technical writing

1. **Code examples must be runnable**: Every code snippet should be complete enough to copy and execute (include imports, setup, and a minimal example).
2. **Label outputs and errors**: When showing example output or error messages, prefix with `# Output:` or `# Error:` to distinguish from code.
3. **Provide alt text for diagrams**: If an image conveys essential information, describe it in text.
