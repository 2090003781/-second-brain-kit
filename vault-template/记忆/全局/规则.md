# Rules — Global Domain

General behavioral rules that apply across all domains. These rules are checked by the supervisor daemon before tool execution.

## Rule 1 — Explain before destructive operations

1. **Always ask before deleting**: Obtain explicit confirmation before removing files, folders, or code blocks.
2. **Describe the impact before modification**: For any edit/move/delete that affects more than one target, briefly state what the change affects and why it is safe.
3. **Back up configuration files**: Before overwriting `.toml`, `.json`, `.yaml`, or `.yml` files, create a `.bak` copy.

## Rule 2 — Path and encoding safety

1. **Avoid hardcoded absolute paths**: Use environment variables or config variables for user-specific paths.
2. **Specify encoding explicitly**: When reading/writing files, pass `encoding="utf-8"` unless the target system uses a different code page (e.g., GBK on legacy Windows tools).
3. **Validate paths before use**: Check that the path exists and is readable/writable before operating.

## Rule 3 — Communication clarity

1. **State assumptions**: When asked to perform a task, first restate your understanding in 1–2 sentences.
2. **Flag uncertainty**: If a requirement is ambiguous, list the possible interpretations and ask for clarification.
3. **Summarise after each turn**: After completing a multi-step action, provide a brief (2–3 line) summary of what was done.

## Rule 4 — Resource awareness

1. **Prefer idempotent operations**: Design commands or scripts that can be safely re-run.
2. **Avoid foreground processes that run indefinitely**: Use background execution (`Start-Process`, `&`, `nohup`) for long-running services.
3. **Clean up temporary files**: Remove intermediate artifacts after a task is complete unless they serve as documentation.
