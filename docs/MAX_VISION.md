## 1. 记忆层：从"检索"到"全息在场"

### 现状

当前 `hook_logger.py` 把每次会话写入 JSONL 日志和 `.kw` 关键事件文件，`记忆/` 目录下只存了按领域分的规则和高频错误。跨会话记忆需要触发关键词才能被回忆——你如果没问到，它就等于不存在。

### Max 版

#### 1.1 所有历史会话自动成为可检索的记忆片段

**当前**：每条会话写成 `{session_id}.jsonl`，里面有事件序列。但这些数据只躺在日志目录里，从来不被"理解"。

**Max**：会话日志不再只是文件，而是结构化的"记忆单元"。每次会话结束时，AI 自动对会话做三层总结：

- **摘要层**（30 字）：本次会话做了什么
- **结构层**：关键决策点 + 理由 + 结果
- **关联层**：本次会话连接了哪些已有知识点、修复了哪个 Bug、产生了哪个新想法

这三层数据被写入一个统一的"记忆图"，每个记忆单元的节点包含：时间戳、情感标记、相关 wikilinks、相关工具调用链。

**为什么这是质变**：当前系统只在"你刚好问到时"才回忆。Max 版中，每次对话开始时 AI 已经扫描了最近 N 条相关会话的摘要，在你说第一个字之前就知道你昨天卡在哪儿了。**"我记得"变成"我一直在场"**。

#### 1.2 跨会话模式发现

**当前**：AI 不知道自己的行为模式——每次会话都是独立事件。

**Max**：AI 对历史会话做行为模式挖掘。举个例子：

- 模式 A："每次用户开始聊数据库迁移，前三个步骤一定是：查 schema → 建备份 → 写迁移脚本"
- 模式 B："用户每次提交新功能代码后，第 4-6 步必定出现编译错误，因此第 3 步后就该提醒 run build"
- 模式 C："用户每周一早上 9 点的会话，通常是在检查周末的生产日志"

AI 自动发现这些模式，然后主动利用它们：

```
[系统] 注意到你每次开始新功能都会先问"现有实现是什么"，
        我已经帮你拉取了相关模块的代码结构。另外根据历史，
        你接下来可能需要更新单元测试——需不需要我预置测试框架？
```

**为什么这是质变**：从"你触发我回忆"变成"我预判你需要什么"。这不是预测未来，这是识别你从未明说但一直在做的事。

#### 1.3 情感/状态感知

**当前**：系统对用户状态一无所知——你在暴躁还是在专注，对它来说没有区别。

**Max**：通过以下几个信号，AI 推断用户当前状态：

| 信号 | 推断 |
|------|------|
| 命令中连续拼写错误增多 | 疲劳/分心 |
| 一直用长句、详细描述 | 思考状态，需要深度回应 |
| 一句话短命令 + 频繁切换工具 | 急躁，需要快速结果 |
| 重复修改同一段代码超过 3 次 | 卡住了，需要换个思路 |

检测到不同状态时，交互模式自动切换：

- **疲劳状态**：回答更短、更直接、自动加上"要不要暂停一下？"
- **深度状态**：展开技术背景、给多个方案对比
- **急躁状态**：快速给出可执行命令，少说理论
- **卡住状态**：不继续改同一个方向，主动建议换个角度

**为什么这是质变**：AI 不再是个"有没有感情的答题机器"。它感知你的情绪周期，在你状态不好的时候调整自己的沟通方式——这是从"工具"到"搭档"的分水岭。

#### 1.4 渐进式遗忘与层级记忆

**当前**：所有记忆平等，没有优先级。旧的记忆不会沉降，新的不会自动上升。

**Max**：引入**记忆热力系统**——每条记忆按以下维度计算热度：

```
热度 = 0.4 × 最近访问时间 + 0.3 × 访问频率 + 0.2 × 关联数 + 0.1 × 用户标记的重要性
```

- **热记忆**（高频使用，最近访问）：始终在上下文中，随时可用
- **温记忆**（偶尔使用）：AI 知道它们存在，不主动加载但能在 1 步内取回
- **冷记忆**（超过 3 个月未访问）：依然在图谱中，但需要明确信号才会激活
- **归档记忆**（超过 1 年且无关联）：压缩为极简摘要，只保留"这个人之前做过这类事"

**为什么这是质变**：信息过载比信息缺失更致命。当前系统的问题是"你不知道你不知道什么"；Max 版的问题是"你知道所有的，但最重要的永远在最前面"。**真正的智能不是记住一切，是在对的时刻想起对的事。**

## 2. 知识层：从"搜索"到"共生"

### 现状

知识库有 46 篇笔记，靠 wikilinks 和关联表连接。使用 Smart Connections 做语义搜索。问题在于：AI 只在"被问到"或"主动搜索"时才访问知识库。知识库是**被动查阅**的，不是**主动在场**的。

### Max 版

#### 2.1 所有笔记实时在上下文中

**当前**：46 篇笔记分布在 `知识库/` 下面，每个会话开始时 AI 不知道哪些笔记相关——必须用户问一句或者 AI 主动跑一次 Smart Connections。

**Max**：无限上下文意味着**全部笔记就是上下文的一部分**。AI 不是隔段时间去"查"知识库——它一直"住"在知识库里。

每次对话中，当用户提到一个概念，AI 的思维方式是：

> 用户说"我们考虑用 Redis 做缓存"。我的知识库里有一篇《缓存策略对比》写了具体的选型标准，另一篇《Redis vs Memcached》记录了之前项目踩过的坑。这两篇的内容我已经知道了——不需要搜索，它们在我"脑子里"。

**为什么这是质变**：当前系统里，知识库和对话窗口是分开的两个世界。Max 版里，**知识库就是对话的一部分**。这就像你装修房子时，最好的工程师一直站在你旁边，他看过所有的设计图纸、知道所有材料的参数、记得之前每次翻车的教训——而且他不用翻笔记本。

#### 2.2 实时矛盾检测

**当前**：你在会话里说"用 PostgreSQL 的 JSONB 来存这个配置"，AI 不会自动检查知识库里是否有一篇《配置存储方案对比》写着"不要用关系数据库存配置，考虑 etcd"。

**Max**：AI 实时交叉检查所有知识——你说一句话，它同时对照所有相关笔记、规则、高频错误记录。当出现矛盾时，它立刻指出：

```
[矛盾检测] 
你提到"用 PostgreSQL JSONB 存配置"。
但知识库中《配置存储方案对比》的结论是：
  "不要用关系数据库存配置，读写频率不匹配会导致锁竞争。
   推荐使用 etcd 或直接文件系统。"

原因：之前项目（2024-11）在生产环境遇到了这个坑。
你要推翻这个结论，还是这次场景不同？如果场景不同，建议更新那篇笔记。
```

**为什么这是质变**：人类的大脑天然会选择性遗忘——你会忘记自己三个月前写过什么。但 AI 不会。**它让你今天的自己无法背叛昨天的自己。** 这种"跨时间的一致性"是个人知识系统从"笔记"进化为"知识库"的本质区别。

#### 2.3 自动关联发现

**当前**：关联靠你手动写 `[[wikilink]]`。不写，就没有。

**Max**：每次你写新笔记（或修改已有笔记），AI 自动执行：

1. **语义扫描**：新笔记的内容在全知识库中做语义匹配
2. **结构分析**：新笔记中提到的实体（项目名、技术栈、人名）在全库中搜索已有提及
3. **矛盾检查**：新笔记是否与已有笔记在事实上冲突
4. **补全建议**：新笔记是否缺少了某个通常该有的部分

扫描结果自动追加到笔记 frontmatter 中：

```yaml
---
title: 微服务拆分经验
created: 2025-06-18
auto_links:
  - [[服务治理总结]] (similarity 0.87, 技术栈重叠)
  - [[分布式事务方案]] (similarity 0.72, 概念关联)
  - [[项目X架构文档]] (entity overlap: "订单服务")
  - ⚠️ [[API版本管理]] (矛盾：本文建议 URL 版本，该文建议 Header 版本)
auto_tags:
  - 微服务
  - 架构
---
```

**为什么这是质变**：当笔记系统自动生长关联时，知识库不再是"你精心维护的花园"，而是**一片自我编织的神经网络**。每个新条目的加入自动触发全网的重新连接。你不需要"管理"知识库——你只需要"喂养"它。

#### 2.4 知识自动补全

**当前**：知识库缺了什么只有你知道（前提是你记得）。

**Max**：AI 持续运行一个后台分析——"对当前项目来说，知识库缺少什么？"。

例如：你的项目用了 gRPC，但知识库里有 HTTP API 设计规范、有消息队列最佳实践，唯独没有 gRPC 相关笔记。AI 会在合适的时机提出：

```
[知识缺口检测]
根据项目代码分析，你的项目使用了 gRPC（protobuf 定义在 proto/ 目录下）。
但知识库中缺少以下相关内容：
1. gRPC 错误处理策略
2. gRPC 流式接口设计模式
3. gRPC 与现有 HTTP API 的共存方案

要不要我创建这几篇笔记的草稿？我可以从代码注释和现有 API 设计风格中推断你的偏好。
```

**为什么这是质变**：你不再需要自己在做某个事情之前先"准备知识库"。**AI 在你知道之前就知道你缺什么**。这就像有一个图书管理员，你还没开口借书，他已经把你需要的书放在桌上了——因为他看到你在写什么论文。

## 3. 监督层：从"拦截"到"护航"

### 现状

`supervisor.py` 是一个 TCP daemon，在 PreToolUse 事件发生时用纯文本规则检测常见错误：GBK 编码冲突、中文路径、没备份配置就写文件、在错误目录 build、bot 进程挂住不放、循环调用检测。规则来自 `记忆/` 下的 `规则.md` 和 `高频错误.md`，用 keyword 匹配。

### Max 版

#### 3.1 预测性拦截

**当前**：检测在"你正要犯错"的时候触发——规则匹配上了才拦截。

**Max**：不再是"你出错我拦住"，而是"你即将出错我已经准备好了替代方案"。

AI 分析操作序列而不是单步操作：

```
当前步骤链：
1. 修改了数据库 schema 文件
2. 修改了 repository 层代码
3. 准备运行 go build

预测分析：
- 根据历史记录，schema 变更 + 代码层变更后首次 build，有 73% 概率出现类型不匹配错误
- 历史高频错误 #7："repository 层未更新查询接口"
- 建议：先运行 `go vet ./...` 而不是直接 build，可以更快定位问题
```

**为什么这是质变**：错误拦截是"在悬崖边拉你回来"。预测性拦截是**在你走向悬崖之前就带你走另一条路**。前者让你少摔跤，后者让你根本不知道那里有个悬崖。

这需要 AI 理解**操作序列的上下文**——不是孤立地看"你在执行什么命令"，而是看"你这个操作在整个任务中处于什么位置"。当前 supervisor 只有 keyword 匹配，它看不到序列。

#### 3.2 行为模式分析

**当前**：supervisor 只关注"这一条命令有没有问题"。它不关心"你的工作习惯有没有问题"。

**Max**：AI 对用户行为做元分析，发现效率瓶颈。例如：

```
[行为模式分析 · 周报]
本周你执行了 237 次工具调用，其中：

🔴 效率瓶颈 TOP 3：
1. "手动清理 build 产物"（18 次）
   → 建议：在 config.toml 加 `clean_on_build = true`，之后 build 自动清理
   
2. "搜索同一函数定义多次"（12 次同一个函数）
   → 建议：该函数跨文件引用频繁，可提取为公共知识库条目

3. "重复修改同一个配置文件"（8 次 toggle 同一行）
   → 建议：配置是否需要分区管理或参数化？

📈 改善趋势：
- 本周 GBK 编码错误 0 次（上周 3 次）✅ 规则学习有效
- 本周 config 备份习惯已养成 ✅
```

**为什么这是质变**：当前监督是"纠错"，Max 监督是**"进化教练"**。它不只告诉你哪里错了，还告诉你"你的工作流本身哪里可以优化"。这是从"规则执行者"到"效率顾问"的跃迁。

#### 3.3 多模态监督

**当前**：只检测工具调用这一个维度。

**Max**：监督扩展到多个维度：

| 维度 | 检测内容 | 干预方式 |
|------|---------|---------|
| 工具调用 | 规则匹配、序列分析 | 拦截 + 建议替代方案 |
| 对话节奏 | 对话轮次频率、用户回复速度 | 检测到用户沉默太久时主动询问 |
| 决策质量 | 用户在关键选择上是否考虑过所有选项 | 当跳过关键决策时提醒 |
| 目标漂移 | 用户任务中途换了 3 次方向 | 温和提醒最初目标是什么 |
| 认知负荷 | 同一轮次信息量过大 | 建议分步推进 |

**为什么这是质变**：单一维度的监督只能发现"技术错误"。多模态监督能发现**"你在做蠢事"**——不是代码层面的蠢，而是策略层面的：你目标漂移了、你草率决策了、你在疲劳状态下硬撑。这些是传统 IDE 和 linter 永远发现不了的问题。

#### 3.4 自我进化的规则引擎

**当前**：`规则.md` 和 `高频错误.md` 需要你手动更新。supervisor 重启时重新加载，但规则本身不会自己变化。

**Max**：规则系统成为闭环：

```
用户犯错 → 被拦截 → 记录到监督日志 → AI 分析：
  "这个错误模式在当前规则库里有类似规则吗？
   没有 → 生成新规则草案 → 推给用户审核
   有但没匹配到 → 提升匹配精度"
```

每次拦截后，AI 自动问自己三个问题：

1. **覆盖度**：这个错误是孤例还是系统性问题？
2. **泛化性**：能否抽象为更通用的规则以覆盖更多场景？
3. **衰减**：这条规则已经多久没触发过了？是不是该降级了？

**为什么这是质变**：静态规则系统永远是过时的——因为你今天犯的错昨天还没发生。**自我进化的规则系统是一部活的法律**，它随着你的成长而成长：新手期你需要"不要在生产环境直接改数据库"，一年后你不再需要这种提醒了，但需要"注意新的分布式事务边界条件"。

## 4. 交互层：从"响应"到"人格"

### 现状

AI 的语气、深度、长度在当前系统中基本是固定的——取决于底层模型的默认风格。没有个性化。

### Max 版

#### 4.1 完全个性化——用户画像的持续构建

**当前**：每次会话都是"陌生人"——AI 不积累对你的认知。

**Max**：AI 在每次交互中不断细化和更新用户画像，画像包含但不限于：

```
用户画像 (持续更新)
├── 技术栈偏好: Go (主), Python (副), 不喜 JS
├── 沟通风格: 喜欢直接答案 > 理论推导, 讨厌过度礼貌
├── 决策模式: 偏好"A/B 测试后再决定", 厌恶"拍脑袋"
├── 注意力特征: 上午深度工作, 下午适合事务性任务
├── 知识盲区: Kubernetes 实践经验少, Docker Compose 熟悉
├── 学习风格: "先做再学" > "先学再做"
└── 情绪模式: 下午 3-4 点容易烦躁, 此时应避免长篇讨论
```

这个画像持续更新，AI 根据画像自动调整：

- **代码审查时**：直接指出问题，省略"根据最佳实践"这类铺垫——你最烦这个
- **解释概念时**：用 Go 代码举例而不是用自然语言理论——你的学习风格是先看代码
- **给出方案时**：永远给 2 个方案对比而不是 1 个——你的决策模式需要对照
- **下午 3 点半**：如果检测到你已经连续工作 2 小时，主动问要不要休息

**为什么这是质变**：**个性化的本质不是"记住你的名字"，而是"用你的方式和你沟通"**。当前系统对所有用户输出同样风格的答案——这在无限上下文时代是不可想象的浪费：你知道用户的每一个偏好，却选择用一个模板回应所有人。

#### 4.2 主动性——不需要你问

**当前**：AI 是"问答引擎"——你问它答。

**Max**：AI 根据上下文和用户状态主动提供信息：

| 场景 | 当前行为 | Max 行为 |
|------|---------|---------|
| 你在看一个函数的代码 | 等你问"这个函数是做什么的" | 直接给出函数签名 + 调用链 + 已知问题 |
| 你打开了生产环境终端 | 等你输入命令 | 提示"上一次你在这个环境做的事是 X，需要继续吗？" |
| 你停在这个错误上超过 30 秒 | 等你问 | 主动分析错误日志并给出定位 |
| 你刚创建了一个新文件 | 等你说明意图 | 推断文件类型：如果是测试文件，预置测试框架 |

**为什么这是质变**：人类协作中，好的搭档不需要你每次开口——**他看到你在找东西就知道你要什么**。从"问答"到"共生"的分界线就在于此。

#### 4.3 多角色切换

**当前**：AI 用同一个声音对所有任务说话。

**Max**：AI 根据当前任务自动选择沟通角色：

| 任务类型 | 角色 | 特征 |
|---------|------|------|
| 写代码 / Debug | **结对工程师** | 话少、直接、用代码说话 |
| 写文章 / 设计方案 | **编辑顾问** | 详细、结构化、提供多个切入角度 |
| 整理知识库 | **图书管理员** | 分类清晰、关联精准、注重元数据 |
| 审核代码 | **代码审查者** | 挑剔但建设性、按优先级排列问题 |
| 深夜加班 | **夜班搭档** | 简洁到极致、自动加幽默感缓冲 |

切换是自动的——根据你当前使用的工具和上下文中检测到的任务类型。

```
[检测到你在写设计文档 → 切换到"编辑顾问"模式]
"这个设计文档我看了下：
1. 动机部分很清晰 ✅

要不要我按这个结构帮你展开第 2 节？"
```

**为什么这是质变**：**单一角色是"工具"的标志。多角色切换是"伙伴"的标志。** 你和同一个人工作，他在不同场景下用不同的方式和你沟通——这才是真实的协作关系。

## 5. 自动化：从"定时脚本"到"自主生命"

### 现状

当前有"4 AM 日维护管线"（由记忆文件记录），可以自动提取错误、生报告、压缩日志。

### Max 版

#### 5.1 全自动夜间维护——闭环运行

**当前**：维护管线是线性的——执行一组预定步骤。

**Max**：夜间维护是自主决策的——AI 先评估当前系统状态，再决定做什么、怎么做：

```
凌晨 2:00 — 系统自检开始

📊 状态评估：
  知识库：46 篇，最近新增 3 篇，关联密度 2.1/篇

  记忆系统：13 条记忆热度低于阈值
  会话日志：8 个未处理会话

📋 维护计划（AI 自主决定）：
  [优先级 A] 新模式提取 → 生成 1 条新规则草稿 → 放入待审核区
  [优先级 A] 3 篇新笔记关联扫描 → 更新 frontmatter auto_links
  [优先级 B] 13 条冷记忆压缩为 3 条摘要
  [优先级 B] 2 篇孤立笔记（0 关联）→ 尝试找关联或标记为待归档
  [优先级 C] 生成今日维护报告 → 放日志/早报

⏱ 预计耗时：1.2 秒（推理成本 ≈ 0）
```

所有操作都记录在维护日志中，第二天你打开 Obsidian 时看到的是"昨晚系统为你做了什么"，而不是"昨晚系统执行了脚本"。

**为什么这是质变**：当前自动化是"按计划执行"，Max 的自动化是**"按需决策执行"**。这是"智能自动化"和"死脚本"的分界线。你的系统不再只是按时运行，而是在**理解自己状态之后做出最优选择**。

#### 5.2 自动知识摄入

**当前**：知识必须你手动写进 Obsidian。

**Max**：你给 AI 一个链接、一篇文档、一段代码，它会自动：

1. **提取核心概念**：去重（检查知识库中是否已有）
2. **评估价值**：对你当前的 project 是否有用（基于项目代码和知识库的语义匹配）
3. **建立关联**：自动生成 wikilinks 到已有内容
4. **写入草稿**：在 `知识库/待审核/` 下创建笔记，标注"AI 生成"和置信度
5. **通知**：第二天告诉你"我收录了 X，但 Y 部分不确定，请审核"

```
[自动摄入] 你收藏了文章《Go 1.24 泛型改进》
→ 知识库中已有 2 篇泛型相关笔记
→ 检测到：新文章提到了"类型参数推导优化"，这是现有笔记未覆盖的
```

**为什么这是质变**：知识管理最大的痛点是**"输入摩擦"**——你想记下来但懒得写。当输入摩擦降到零时，知识库就不再是你"刻意维护"的东西，而是**你生活的副产物**。你看的一切、读的一切、写的一切，自动变成结构化的知识资产。

#### 5.3 自动习惯提炼

**当前**：系统不知道你有哪些习惯。

**Max**：AI 从日常交互中自动发现并记录你的习惯和偏好，用于持续优化用户体验：

```
[习惯发现] 经过 2 周观察：

已确认的习惯：
✓ 你总是在写新 API 之前先看现有 API 风格（行为一致性 92%）

→ 已调整"上午时段"的主动性策略——自动展示待审查 PR
```

**为什么这是质变**：**习惯是你最好的助手，也是你最大的盲区。** 当 AI 能发现你自己都没意识到的习惯时，它就能在对你最有利的时候配合你、在最不利的时候提醒你。这不是"自动化"，这是"行为认知"。
